## 二、AI 给出过但我没有采纳的建议

### 2.1 内存队列设计（未采纳）

**AI 建议**：
- 使用 `LinkedBlockingQueue` 作为主队列
- 失败任务立即重新入队
- 消费线程直接从队列取任务

**未采纳原因**：
- 内存队列满会丢任务
- 服务重启会导致未处理任务丢失
- 失败任务会抢占新任务的资源

### 2.2 企业级架构（未采纳）

**AI 建议**：
- 引入 Redis Stream 作为消息队列
- 使用 Nacos 作为配置中心
- 引入 Prometheus + Grafana 监控

**未采纳原因**：
- MVP 阶段不需要这么复杂
- 增加运维成本
- 违背奥卡姆剃刀定律

## 三、哪些关键决策是我自己做出的

### 3.1 采用 MySQL 拉取模式

**决策**：分配线程持续从 MySQL 拉取任务到内存队列，工作线程池执行。

**原因**：
1. **可靠性**：任务持久化在 MySQL，不会丢失
2. **简单性**：不需要复杂的消息队列
3. **可控性**：数据库作为消息源，便于监控

### 3.2 新任务和重试任务分离

**决策**：新任务和重试任务完全分离，使用独立的拉取逻辑和队列。

**原因**：
1. **避免阻塞**：大面积失败时，新任务不会被hang住
2. **冷却机制**：重试任务有冷却时间，避免频繁重试
3. **优先级保证**：新任务永远优先处理

### 3.3 重试冷却机制

**决策**：通过数据库 updated_at 时间控制重试冷却，而不是使用 DelayQueue。

**原因**：
1. **实现简单**：不需要额外的延迟队列
2. **数据库控制**：利用 MySQL 的 updated_at 自动更新
3. **可配置**：冷却时间可通过参数调整

### 3.4 简化状态流转

**决策**：只使用 PENDING、SUCCESS、FAILED 三种状态。

**原因**：
1. 不需要 DELIVERING 状态（简化判断逻辑）
2. 失败任务保持在 PENDING 状态
3. 更容易理解和维护

## 四、工程判断依据

### 4.1 可靠性优先

> **"任务不能丢，这是底线。"**

在设计系统时，我把可靠性放在第一位：
- 使用 MySQL 作为消息源，而不是内存队列
- 服务重启后，未处理的任务会自动被拉取

### 4.2 隔离设计

> **"新任务和重试任务应该隔离。"**

这是我在审阅 AI 方案后主动提出的改进：
- 避免了失败任务抢占新任务的资源
- 通过冷却机制控制重试频率
- 保证了业务系统的请求不会被阻塞

### 4.3 演进思维

> **"MVP 先跑起来，未来按需扩展。"**

当前设计：
- 单节点部署
- MySQL 作为消息源
- 未来可以迁移到 Redis/RabbitMQ

- ✅ 提供技术选型参考
- ✅ 生成代码模板

### 5.2 我的判断

- ✅ 根据实际场景调整
- ✅ 主动提出新任务/重试任务分离的改进

> **MySQL 作为消息源 + 新任务/重试任务分离 + 工作线程池统一消费**
>
> **核心优势**：任务不丢失，新任务不被阻塞，重试冷却可控
